# Why do we create embeddings for both the document and the question?

Both the document chunks and the user's question are converted into embeddings so they exist in the same vector space.

This allows ChromaDB to compare their meanings using semantic similarity instead of comparing raw text.

Without embeddings, the computer cannot understand or compare the meaning of text.

# model.encode(chunks)

When `model.encode()` is given a list of chunks, it generates one embedding for each chunk.

Example:

```python
chunks = [
    "Chunk 1",
    "Chunk 2",
    "Chunk 3"
]

embeddings = model.encode(chunks)
```

The output is a list of embeddings.

```text
[
    [384 numbers],   ← Chunk 1
    [384 numbers],   ← Chunk 2
    [384 numbers]    ← Chunk 3
]
```

Each chunk has its own 384-dimensional embedding.

This allows every chunk to be stored separately in ChromaDB for semantic retrieval.


# Designing the Retrieval Pipeline

Instead of writing all the code in one place, the RAG pipeline is divided into small functions.

Each function performs one specific task.

Example:

```text
extract_text()
↓
split_text()
↓
create_embeddings()
↓
store_in_chromadb()
↓
retrieve()
```

This makes the code easier to understand, debug, maintain, and reuse.


# Required Libraries

To build the retrieval pipeline, we need different libraries for different tasks.

| Library | Purpose |
|----------|---------|
| PyMuPDF (`fitz`) | Extract text from PDFs |
| RecursiveCharacterTextSplitter | Split large text into chunks |
| SentenceTransformer | Generate embeddings |
| ChromaDB | Store embeddings and perform semantic search |

These libraries together form the retrieval component of a RAG system.


# Loading the Embedding Model

The Sentence Transformer model is loaded using:

```python
model = SentenceTransformer("all-MiniLM-L6-v2")
```

Loading the model does not create embeddings.

It loads the pretrained embedding model into memory, making it ready to convert text into embeddings using:

```python
model.encode(text)
```

The model `all-MiniLM-L6-v2` generates a **384-dimensional embedding** for each input sentence or chunk.


# Opening a PDF

A PDF is opened using:

```python
doc = fitz.open(pdf_path)
```

`pdf_path` is the path of the PDF file.

`fitz.open()` loads the PDF into memory and returns a **Document object**.

The Document object allows us to:

- Read pages
- Count pages
- Extract text
- Close the PDF

# Extracting Text from a PDF

```python
doc = fitz.open(pdf_path)

text = ""

for page in doc:
    text += page.get_text()
```

`text` is initialized before the loop so that text from every page is accumulated into a single string.

If `text = ""` were placed inside the loop, it would be reset during every iteration, causing only the last page's text to be stored.


# Returning the Extracted Text

After extracting text from every page, the PDF should be closed using:

```python
doc.close()
```

This releases the resources used by the opened PDF.

The extracted text is returned using:

```python
return text
```

This allows other parts of the program to use the extracted text for chunking and embedding generation.


# Generating Embeddings

Embeddings are created using the Sentence Transformer.

```python
embeddings = model.encode(chunks)
```

When `model.encode()` receives a list of chunks, it generates one embedding for each chunk.

Example:

10 Chunks

↓

10 Embeddings

Each embedding generated by `all-MiniLM-L6-v2` has **384 dimensions**.

These embeddings are later stored in ChromaDB for semantic retrieval.


# Storing Data in ChromaDB

After generating embeddings, the chunks are stored in ChromaDB.

Each chunk requires:

- A unique ID
- The document (chunk text)
- The embedding
- Metadata

```python
collection.add(
    ids=ids,
    documents=chunks,
    embeddings=embeddings.tolist(),
    metadatas=metadatas
)
```

Since `model.encode()` returns a NumPy array, `tolist()` is used to convert it into a Python list before storing it in ChromaDB.
